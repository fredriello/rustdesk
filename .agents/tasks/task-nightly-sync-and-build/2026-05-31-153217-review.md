# Upstream sync before nightly build

The change introduces a reusable workflow (`sync-upstream.yml`) that merges upstream `rustdesk/rustdesk` master into the fork's master branch, and wires it as a prerequisite to the existing nightly build workflow. The sync runs daily at 22:00 UTC — two hours ahead of the midnight nightly — and can be invoked standalone or as a called workflow. The nightly build now gates on a successful sync before invoking `flutter-build.yml`.

**Watch for:** The sync job pushes to the `master` branch, but the nightly workflow itself triggers on the default branch (`terminal-utf8-and-reconnect` based on the remote HEAD). If this fork's default branch isn't `master`, the sync will update a branch that the nightly build may not be running against (confirmed). The `GITHUB_TOKEN` push lacks protection against racing with other pushes or concurrent schedule triggers (possible).

## High-level view

The sync workflow hardcodes `master` as both the checkout ref and the push target. If the fork's default branch is something else, the cron-triggered nightly build will execute against the default branch — not `master` — creating a split where the sync and build operate on different branch tips.

The `GITHUB_TOKEN` can push to the repository's own branches but cannot push to protected branches unless protection rules explicitly allow GitHub Actions. If `master` has branch protection, the push step will fail at runtime with a 403.

The sync also runs on its own cron (22:00 UTC) in addition to being called by the nightly (00:00 UTC). The midnight invocation is redundant if the 22:00 run succeeded — it will find nothing new to merge. Removing the standalone schedule from `sync-upstream.yml` would simplify without losing functionality.

## Branch targeting and the default-branch gap

The sync workflow hardcodes `master` everywhere:

```yaml
ref: master
...
git push origin master
```

Meanwhile, the repository's default branch (based on `origin/HEAD`) is `terminal-utf8-and-reconnect`. GitHub Actions cron triggers run against the default branch. This means the nightly workflow file that executes is the one on `terminal-utf8-and-reconnect`, but the sync job inside it pushes fresh upstream code to `master`. Unless the build also explicitly checks out `master`, the nightly artifacts will be built from the default branch tip — which may be stale relative to the sync.

If the intent is that `master` *is* the primary branch and `terminal-utf8-and-reconnect` is temporary, this will sort itself out once the default branch is switched back. But as currently configured, the two workflows operate on different branch targets.

## Concurrency and race conditions

The sync workflow has no `concurrency` key. Given the dual-trigger setup (its own cron at 22:00 and `workflow_call` from the nightly at 00:00), overlapping runs are unlikely under normal operation. However, manual `workflow_dispatch` triggers during a scheduled run could produce two instances attempting `git push origin master` simultaneously. The second push would fail with a non-fast-forward error since it checked out `master` at the same commit as the first but the first already pushed a merge commit.

Adding a concurrency group would prevent this:

```yaml
concurrency:
  group: sync-upstream
  cancel-in-progress: true
```

## Merge conflict behavior

The merge step fails closed: conflicts abort the merge, emit a `::error::` annotation, and exit non-zero, which blocks the downstream nightly build. This is the right failure mode for an unattended sync — no silent broken merges.
